# 🔥 一、总战略（2026-05 重规划版）

> ✅ **SEO流量系统 + 多子站系统化试跑 + 主站转化系统**


## 👉 核心模型

```text
1 个主站承接中心（/games/ + /boxes/ + /rank/ + /01zhe/）
+
N 个可复制、可淘汰的候选子站
每个子站前期只守 1 个小类目
↓
每个子站内部都按“7 仓位起步”
A-G 仓位分别承接 7 个游戏 / 7 组搜索意图 / 7 条内容词路
每个仓位独立管理：关键词池、后台标签、内链入口、数据面板
对外 sitemap 仍按站点原有板块 / 分类输出，仓位只在后台做赛马观察
↓
所有子站从首批页面开始统一导流主站 /games/[id] 或 /boxes/
↓
先看站内 14~21 天哪些仓位先起势
↓
再看站级 30~60 天哪个子站该扩量、冻结、淘汰或补位
↓
赢家子站先扩当前小类目的更大词和更多游戏
↓
当前类目打透后，再并入相邻品类，逐步把赢家子站做成更综合的全品类子站
↓
全部子站持续向主站导流，主站负责最终转化
```

> 💡 **本次调整的关键变化**：
> - 项目层不是“只有 7 个仓位”，而是 **多子站矩阵**。
> - **7 仓位**是每一个子站内部的标准起步模板，不是整个项目只有 7 个试验位。
> - 每个子站前期只做 1 个小类目；起量后再扩更大词、更多游戏、再扩相邻品类。
> - 不行的是整站淘汰换新站，不是放弃整个多子站模型。

> 💡 **核心分工原则**：
> - **主站 = 转化中心（产品站）**：承接所有子站流量，完成盒子下载/注册转化。
> - **子站矩阵 = 外层试跑系统**：负责小类目验证、整站淘汰补位、系统化复制。
> - **子站内 7 仓位 = 内层起跑模板**：负责问题词起量、索引验证、站内赢家筛选。
> - **资源分配 = 双层数据驱动**：先看站内谁冒头，再看哪个子站值得继续投。

> 最终实现：**多子站并行 → 单站 7 仓位起跑 → 赢家子站扩词扩游戏 → 全部导流主站转化**。

## 子站模板路由结构（适用于多子站实例）

> 子站不再定义为“唯一的深度攻略站”，而是定义为“可复制的 SEO 试跑模板”。
> 一个子站前期 = 一个小类目 / 一组相近搜索意图 / 6~12 款关联游戏；站内再拆成 7 个起步仓位。
> 先跑出站内赢家和子站级 Impression，再决定是否继续加码；`Next-web-sub` 负责提供多站母版。
> `/games` 和 `/boxes` 路由继续通过生产环境配置变量控制是否开启：
> - 默认关闭（推荐）：子站只保留内容页、聚合页、Tag页
> - 若开启：页面仅作内容展示，canonical 指向主站，不显示下载/购买按钮

```
首页 /
│
├── /guides/             ← 攻略列表（SEO主力，顶级路由）
│   └── /guides/[slug]   ← 攻略详情（含 → 主站游戏页 内链）
│
├── /reviews/            ← 评测列表（多游戏横评）
│   └── /reviews/[slug]
│
├── /news/               ← 版本资讯 / 新卡发布
│   └── /news/[slug]
│
├── /tag/[tag]/          ← 按游戏/主题聚合攻略（内链枢纽）
│
│   ── 以下路由受 NEXT_PUBLIC_ENABLE_GAME_ROUTES 控制 ──
│
├── /games/[id]          ← （可关闭）游戏介绍页，canonical→主站，无下载按钮
├── /games/category/[slug] ← （可关闭）品类页，canonical→主站
└── /boxes/[id]          ← （可关闭）盒子简介，canonical→主站，不含推广链接
```

> ✅ 子站首发只做“入口页 + 问题页 + 聚合页”
> ✅ 每个内容详情页末尾强制出现"前往主站体验/下载"引流入口
> ❌ 禁止：子站出现任何 APK 直链、盒子推广点击链接、完整转化闭环
> ❌ 禁止：把子站做成“迷你主站”或“大而全综合站”


# 🧠 三、主站 vs 子站矩阵 战略分工（核心升级🔥）


## 🟢 主站（转化中心）—— `Next-web`（我爱玩游戏网 / 5awyx.com）

```
定位：游戏盒子折扣聚合站（转化中心/产品站）

保留页面：游戏列表页、游戏详情页（下载/盒子）、分类页

        简短 / 服务产品 / 不做SEO主力

```

> ✅ 主站核心是"盒子对比"，文章配合游戏/盒子页面辅助转化，不做深度内容主力


## 🔴 多子站矩阵（项目层试跑体系）

```
定位：多个可复制、可淘汰的小类目子站系统，不是只有 1 个 7 仓位大盘
单站规则：
  1️⃣ 每个子站前期只做 1 个小类目
  2️⃣ 每个子站内部都跑 7 仓位起步模板
  3️⃣ 所有子站从首批页面开始统一导流主站
  4️⃣ 站内先看哪个仓位冒头，站级再看哪个子站值得扩量
  5️⃣ 子站起量后，先扩当前类目的更大词和更多游戏
  6️⃣ 当前类目打透后，再并相邻品类，逐步成长为更综合的赢家子站
  7️⃣ 弱势子站可以整站淘汰，释放资源给新子站
❌ 不做：一个子站上线就打全品类 / 还没验证就横向扩很多品类 / 子站之间互相抢主词
```

### 单个子站的 7 仓位起步模板

```text
一个子站 = 1 个小类目 + 7 个起步仓位
仓位可对应：7 款关联游戏 / 7 组问题词 / 7 条意图链路
每个仓位都要独立管理标题池、内链入口、后台观察标签、数据记录
对外 sitemap 继续按板块 / 分类输出，不把仓位直接暴露成公开信息架构
单站 Weekly Silo：周一 A，周二 B，周三 C ... 周日 G
先跑出站内赢家仓位，再决定整站是否扩量
```

> ✅ `Next-web-sub` 现有代码框架适合作为“多子站母版 + 单站 7 仓位起步模板”；未来是否拆成独立品牌，取决于具体子站是否跑通。


## 👉 一句话理解：

```text
主站：总仓（承接转化）— 不负责内容赛跑
子站矩阵：外层赛马盘（验证哪些小类目站值得继续投）
单站 7 仓位：内层起跑器（验证这个子站里哪条词路先冒头）
```

# 🟢 主站内容结构（转化优先）

> ⚠️ **策略调整**：主站文章只发「功能型内容」，不做深度攻略/教程类SEO文章。
> 深度内容统一在子站发布，主站专注承接流量并完成转化。


> ❗主站内容在这套机制里不是“产能中心”，而是“承接补强层”；预算默认先给子站矩阵，只有承接链路缺页时才补主站文章。

❌ **主站停止发布的内容类型（统一迁移到子站）：**

# 🔴 子站内容结构（多子站试跑版🔥）—— Next-web-sub 模板

> 子站仍走 `/guides/[slug]`、`/reviews/[slug]`、`/news/[slug]`、`/tag/[tag]` 路由。
> **核心策略从「单游戏打穿」改成「单小类目试跑 + 赢家加码」。**
> 一个子站首批只做 10~20 页：1 个站点定位页 + 1~2 个主题/Tag 聚合页 + 6~12 个问题页 + 2~4 个时效/对比页。

```
小类目入口页 20% → guides/tag（定义站点主题）
具体问题页   45% → guides（怎么玩 / 阵容 / 值不值得练 / 怎么获得）
对比/筛选页 15% → reviews/guides（A和B哪个好 / 零氪怎么选）

权威锚点页   5% → reviews（人工实测 / 横评）
```

> ✅ 每个子站只服务一个小主题，不要求一次覆盖整个品类
> ✅ 每篇文章至少链接 1 个同站主题页 + 2 篇相关问题页 + 1 个主站目标页
> ✅ 胜出站才从 20 页扩到 50 页，再扩到 100 页
> ❌ 禁止刚上线就做“百科式大全”或几十篇同模板灌站

# 🔗 五、子站内链系统（试跑版）

------

## 每篇子站文章必须：

```
① → 链回本站首页/主题页（定义站点主题）
② → 链 2~3 篇同主题问题页（形成小闭环）
③ → 链 1 个主站目标页（/games/[id] 或 /boxes/）
④ → 只在强相关时，再链主站 /rank/ 或专题页
```

------

## 内链结构：

```
子站首页/主题页
  ↓
问题页 / FAQ / 对比页
  ↓
主站游戏页 或 /boxes/
```

------

## 各页面内链职责

| 页面 | 指向 | 作用 |
|------|------|------|
| 首页 / 主题页 | 问题页、对比页、版本页 | 定义站点主题，承接入口词 |
| 问题页 | 同主题文章、主站目标页 | 吃长尾词并导流 |
| 对比页 | 问题页、/boxes/ | 承接决策型搜索 |
| 版本页 / 新闻页 | 主题页、问题页 | 提供更新信号和时效性 |

------

👉 本质：

> ❗子站不是为了构建完整闭环，而是为了用最短链路验证“哪个小类目先出头”

# 📈 九、内容生产策略（多子站矩阵 + 单站 7 仓位赛马版）

> 项目层看“哪个子站该继续投”，站内层看“哪个仓位先冒头”。两层赛马同时存在。

------

## 内容预算优先级

1. 先选候选子站，每个子站前期只守 1 个小类目。
2. 每个候选子站内部同步建立 7 个仓位，跑完整的首轮页面集。
3. 所有子站从第一批页面开始就必须带主站导流链路。
4. 先看站内 14~21 天哪些仓位冒头，再看站级 30~60 天哪个子站该扩量或淘汰。
5. 赢家子站先扩当前类目的更大词和更多游戏；类目打透后再并相邻品类。
6. 弱势子站可整站淘汰，资源转给新子站。

## 双层状态模型

### 站级状态（项目层）

- 候选子站（Candidate Site）：前期只守 1 个小类目，先验证这个站值不值得继续做。
- 赢家子站（Winner Site）：当前类目已起量，开始扩更大词、更多游戏和更深内容。
- 输家子站（Loser Site）：整站增长弱、站内 7 仓位长期无人冒头，可冻结或淘汰。
- 新候补子站（New Candidate Site）：替换输家站进入系统，继续沿用同一套 7 仓位模板。

### 站内状态（单个子站内部）

- 起跑仓位（Running Slot）：按 A-G 周循环跑首轮页面集。
- 冒头仓位（Winner Slot）：先拿到 Impression、Query 或更好的 CTR，优先追加人工和深度页。
- 弱势仓位（Weak Slot）：起势慢，但暂不马上删除，保留观察与尾流入口。

## 单个子站的 Weekly Silo

```text
周一：仓位 A，5 篇
周二：仓位 B，5 篇
周三：仓位 C，5 篇
周四：仓位 D，5 篇
周五：仓位 E，5 篇
周六：仓位 F，5 篇
周日：仓位 G，5 篇
```

> 这是单个候选子站的标准起步节奏，不是整个项目总共只能发这 5 篇。

## 单站每天 5 篇到底怎么分（最重要的执行口径）

> 重点不是“5 篇平均分给 5 个游戏”，而是“5 篇先服务当前站点的验证目标”。
> 单站每天的分配要同时看两层：一层看**游戏池配比**，一层看**页面类型配比**。

### 先定义 3 个池子

- 日常游戏池：已经纳入当前站点、需要持续拿数据的核心游戏池。通常是站内已确定的 6~12 款游戏中的主力部分。
- 增量游戏池：同小类目下准备继续扩的候补游戏/候补词路。它们不是完全陌生的新游，但还没进入日常主力池。
- 新游戏测试位：当天只拿很少资源去试的新游戏、新上线词，或刚并入的小候补游戏。它是试错位，不是主生产线。

### 候选子站起步期（前2~3周）

```text
推荐配比：日常游戏池 60% + 增量游戏池 20% + 新游戏测试位 20%
换成 5 篇/日就是：3 篇日常 + 1 篇增量 + 1 篇新游戏测试
```

#### 对应到当天 5 篇的实际排法

```text
1 篇：当前仓位的主题入口页 / 聚合页（算日常盘，因为它服务当前主打词路）
2 篇：日常游戏池的问题页 / 攻略页 / 阵容页
1 篇：增量游戏池的问题页 / FAQ页
1 篇：新游戏测试位的首篇试跑页（首篇问题页 / 首篇对比页 / 首篇截流页）
```

#### 这句话一定要记住

```text
单站每天 5 篇，不是 5 篇都上新游戏。
候选站前期本质上是：3 篇稳盘 + 1 篇扩池 + 1 篇试新。
```

### 候选子站观察期（第4~8周）

```text
推荐配比：日常游戏池 60% + 增量游戏池 40% + 新游戏测试位 0%
换成 5 篇/日就是：3 篇日常 + 2 篇增量 + 暂停每天强上新
```

说明：

- 这时候先把已经证明有潜力的同类目游戏池跑深，不再每天硬塞新游戏。
- 如果确实有强事件型新游，只能临时占用“增量位”，不要额外加量打乱整站节奏。

### 赢家子站加码期（站级确认可扩量后）

```text
推荐配比：日常游戏池 40% + 增量游戏池 40% + 新游戏/更大游戏测试位 20%
换成 5 篇/日就是：2 篇日常 + 2 篇增量 + 1 篇更大词或新并入游戏
```

说明：

- 赢家站不是突然全站改写成“天天追新”。
- 赢家站的 1 个新位，优先给“更大词 / 更大游戏 / 当前类目下最值得并入的新对象”，而不是乱上完全无关的新题材。

### 输家子站 / 冻结观察期

```text
推荐配比：日常游戏池 80%~100% + 增量游戏池 0%~20% + 新游戏测试位 0%
```

说明：

- 这时已经不该继续拿新游戏做试错。
- 目标只剩两个：保留已有收录、观察尾流，等待整站淘汰或低频续命判断。

### 页面类型配比不要变

不管你当天的游戏池怎么切，**页面类型配比仍然尽量保持：**

```text
1 篇入口/主题页 + 3 篇问题/攻略页 + 1 篇 FAQ/对比页
```

也就是说：

- 游戏池配比决定“今天写哪类游戏”
- 页面类型配比决定“今天写什么页面结构”

这两个维度要同时成立，不能只看其中一个。

## 产能规则

```text
单个候选子站：固定 5 篇/日，连续运行 2~3 周，先完成一轮到两轮站内验证
单个赢家子站：先保持完整周循环，再扩更大词、更多游戏；必要时再加第二档期
项目总产能：= 活跃子站数 × 单站节奏；资源不足时减少活跃子站数量，不要把所有子站都压成半死不活
```

## 页面目标

```text
主站：承接链路完整 > 发文数量
单个候选子站：完成 7 仓位首轮验证，并在站内跑出 1~2 个明显冒头仓位
单个赢家子站：先打透当前小类目，再决定是否扩更多游戏、更大词和相邻品类
输家子站：停更观察或整站淘汰，不做无限续命
```

```
  ↓

```

```
  ↓

  ↓
```

# 📊 十二、核心KPI（判断成败）

------

> KPI 优先级顺序：站内仓位起势 > 子站整体起势 > 全部子站导流 > 主站转化。

## 一级KPI：单个子站的 7 仓位是否有人冒头

```text
1. 14~21 天内，是否至少有 1~2 个仓位拿到稳定 Impression
2. 单站 7 仓位整体收录率是否 ≥ 60%
3. 是否出现 5~10 个进入前 50 / 前 20 的查询词
4. 哪个仓位的 Impression / CTR 增速最快
5. 哪个仓位的改稿成本最低、最适合作为当前子站的扩量入口
```

## 二级KPI：子站整体是否值得继续扩量还是淘汰

```text
1. 站内是否已经形成“主题页 → 问题页 → FAQ/对比页 → 主站目标页”闭环
2. 子站总 Impression / Query / 导流点击是否进入明显增长期
3. 当前小类目内是否还有更多游戏和更大词可扩，而不是已经撞到边界
4. 同站扩量是否仍不与主站或其他子站冲突
5. 30~60 天后，这个子站该继续扩量、冻结观察，还是整站淘汰
```

## 三级KPI：全部子站导流与主站承接是否有效

```text
1. 来自全部子站的 /games/[id] 或 /boxes/ 点击是否持续增加
2. winner 子站贡献的导流点击占比是否逐步提高
3. /games/[id] → /boxes/[id] 的点击率是否持续上升
4. 盒子推广点击 / 注册转化是否稳定增长
```

## Winner / Loser 判定

```text
赢家子站（Winner Site）：站内已有 1~2 个明确赢家仓位 + 全站 Impression 增速领先 + 导流质量更高 + 当前类目仍有扩量空间
输家子站（Loser Site）：站内 7 仓位长期都不起量 / 无收录 / 改稿成本高 / 主题边界不清
新候补子站（New Candidate Site）：补位测试新的小类目，继续沿用同一套 7 仓位起步模板
```

------

👉 第一优先看单站 7 仓位里有没有人冒头
👉 第二优先看哪个子站整体值得继续扩量
👉 第三优先看全部子站是否已经把流量送进主站
👉 主站页面总数本身不是胜负指标

# ⚡ 十四（新增）、人工 vs AI 内容分工

## 🛡️ Google HCU（有用内容算法）风险防护

> HCU（Helpful Content Update）的核心判断：**这批内容是为人写的，还是为SEO写的？**
> 批量AI内容的最大风险是"结构高度相似 + 内容可替换"，被整站打标后排名全面下滑。
> 以下机制已内嵌在 [AI提示词工作流.md](./AI提示词工作流.md) 中，执行时必须严格遵守，不可跳过。

------

### 防护机制1：先验证搜索需求，再写文章（Step 0）

```
❌ 错误做法：批量生成标题，无论有没有人搜都发布
✅ 正确做法：每个标题的核心词必须经过 Step 0 关键词挖掘验证
           没有搜索量的关键词 = 不写
           工具：百度搜索框下拉联想 / 百度指数 / 5118 / 爱站
```

> 意义：只产出有真实需求的内容，让每篇文章都有潜在读者，从源头避免"为爬虫写内容"的信号。

------

### 防护机制2：游戏专属痛点，禁止套模板（Step 1）

```
❌ 低质量示例（AI套模板）："这款游戏太肝了，每天要做很多任务"
✅ 高质量示例（绑定真实游戏机制）："每天需完成12个日常任务，公会战强制在线，
   活动期间需连续签到7天" → 导致上班族/轻度玩家流失

判断标准：Step 1 输出的痛点描述中，
          如果没有出现该游戏独有的系统或机制名称，说明AI在套通用模板，该篇不可用。
```

对AI不熟悉的冷门游戏，必须使用「评论辅助版」：从 TapTap/App Store 差评区复制 10~20 条真实评论喂给 AI，提取真实痛点，而不是让 AI 自行发挥。

------

### 防护机制3：每篇核心论点必须不同（Step 4 质量红线）

Step 4 审核发布前，必须逐项通过以下检查：

| 检查项 | 说明 |
|---|---|
| 痛点是否具体 | 文章中的游戏问题描述必须有具体细节，不接受"太肝/太氪"等泛泛描述 |
| 推荐游戏是否子站有 | 不能推荐子站没有收录的游戏，防止死链 |
| 内链占位符是否替换 | 所有 `{{URL}}` 已替换为真实游戏/盒子URL |
| 折扣数据是否真实 | `[折扣:X折待填]` 已替换为主站后台实际价格，不接受AI编造 |
| 核心论点是否差异化 | 同一款游戏的不同角度文章，推荐理由必须不同，不能只换游戏名 |

------

### 防护机制4：发布节奏控制（先质量后放量）

```
阶段0（候选子站起步期，第1~2周）：
  → 每个候选子站先完成 1 轮 7 仓位页面集（按单站 Weekly Silo 跑满）
  → 主站只补承接页，不跟着一起堆内容
  → 不扩新类目，只验证当前小类目和站内仓位是否有起势信号

阶段1（站内冒头期，第3~4周）：
  → 先把站内资源投给冒头仓位，再判断这个子站整体是否值得继续加码
  → 项目总量 = 活跃子站数 × 单站节奏，不强行平均给所有站
  → 每篇仍要过完整审核红线，不得走形式

阶段2（站级加码期，第2个月起）：
  → winner 子站先扩当前小类目的更大词和更多游戏，再考虑并入相邻品类
  → loser 子站冻结或淘汰，换新小类目站补位
  → 主站每周 0~2 篇，只补承接型页面
  → 同步保持人工精品文章每周 3~5 篇（主站承接页 + winner 子站锚点页），作为 E-E-A-T 锚点
```

> ❗切忌在建站第一周就把矩阵总量一下子冲到远超 10 篇/天。Google 对新站的初始评分窗口极为敏感，
> 大量结构相似内容会直接触发"站点级别"质量惩罚，后续很难恢复。

------

### 防护机制5：人工精品内容作为E-E-A-T锚点

```
AI 内容量再大，如果站上没有任何"真人写的、有专业深度的内容"，
Google 会将整站判定为 AI 内容农场。

人工精品文章（guide-section/review-section）的作用：
- 提供真实经验（Experience）和权威性（Authoritativeness）信号
- 让 Google 评估时有"高质量样本"参照
- 每周 3~5 篇，是 AI 批量内容能安全放量的前提条件，不可省略
```

## 16.1 为什么不建议先做一个“大而全子站”

| 维度 | 一个大子站深打 | 多子站试跑 |
|------|---------------|------------|
| 起量速度 | 慢，前期要堆很多内容才看得见信号 | 快，2~3周就能看到哪个小类目先出头 |
| 风险 | 方向选错就是整站浪费 | 输家可冻结，换一个新候选继续跑 |
| 资源利用 | 人工和 AI 都容易砸在错误主题上 | 资源可以向赢家倾斜，效率更高 |
| 关键词边界 | 容易越做越宽，最后和主站重叠 | 每个子站只守一个小主题，边界清晰 |
| 扩张方式 | 先重投入，后验证 | 先验证，后复制 |

**结论：子站不该先按“垂类大站”去建设，而应先按“可淘汰的试跑站”去运营。**

---

## 16.2 子站试跑关键词分层模型

```
层级1：站点主题词（小类目 + 场景）
  → "零氪卡牌手游推荐"、"放置养成卡牌推荐"
  → 靠首页 / 主题页承接，用来定义整个站在讲什么

层级2：游戏 + 问题词（最先起量）
  → "XX游戏怎么玩"、"XX游戏新手阵容"
  → 靠 guides 详情页承接，是试跑初期主战场

层级3：游戏 + 解决方案词
  → "XX游戏抽卡规划"、"XX游戏零氪养成"
  → 靠 FAQ / 教程 / 对比页承接

层级4：时效词
  → "XX游戏赛季卡组"、"XX游戏版本改动"
  → 靠 news / 版本分析页承接，强化更新信号

层级5：大品类词（后置）
  → "2026卡牌手游推荐"、"最好玩的放置手游"
  → 只有胜出子站才接，不在试跑初期碰
```

**试跑初期：80% 精力放在层级2和3，20% 给层级1；层级5 一律后置。**

## 16.5 单个子站的 Technical SEO 执行重点（7 仓位起步版）

### 高优先级（影响爬取、分类识别和索引）

| # | 问题 | 文件位置 | 修复方案 |
|---|------|---------|---------|
| 1 | ❌ sitemap 还没按板块 / 分类稳定输出 | `src/app/sitemap*` / `public/` | 生成 guides / reviews / news / category 等稳定 sitemap，并在 sitemap index 与 robots.ts 中统一声明；仓位只作为后台观察字段 |

| # | 问题 | 文件位置 | 修复方案 |
|---|------|---------|---------|

### 低优先级（优化项）

| # | 问题 | 说明 |
|---|------|------|
| 8 | sitemap 单文件过大 | 改为按板块 / 分类 / 分页拆分，单文件建议控制在 10000 条以内 |
| 9 | 图片 WebP + 懒加载 | Core Web Vitals 优化 |
| 10 | 文章 dateModified | 在 Article JSON-LD 中正确填写 `dateModified` |
| 11 | FAQPage / SoftwareApplication 结构化数据利用不足 | 在 FAQ、盒子说明、工具型页面补足 schema，争取富结果展示 |

---

## 漏洞 #1 🔴 AI内容被 Google HCU/SpamBrain 识别风险

**状态：🔄 解决中**

**核心问题（重新定位）：**
> 问题不是 AI 占比多少，而是 **Google 能否识别出来** + **内容是否真正满足用户意图**。
> 这两个条件同时成立时才会被惩罚——识别出来但内容质量好的站不一定被罚，无法识别的 AI 内容不会被罚。

**Google HCU 实际惩罚的是：**
1. 内容与用户搜索意图不匹配（高跳出率、低停留）
2. 大量文章呈现可识别的模板化"AI指纹"（句式重复、过度补充说明、无原创信息）
3. 文章大量依赖泛化内容，无任何专属于本站的独特数据

**"破解截流型"的真正问题：**
- 用户搜索《XX破解版》→ 想知道有没有能用的破解版
- 文章内容却是《正规盒子才是正确选择》→ 意图完全错位

#### 方案B：内容质量标准（防 HCU 核心检查规则）

> 每篇文章发布前必须过以下标准，AI 草稿不符合的必须修改后才能发布。

**一、内容有效性检查（1~5）**

**✅ 1. 这篇文章真的解决一个问题吗？**

发布前问自己3个问题：
- 用户为什么要搜这个关键词？
- 这篇文章有没有直接回答问题？
- 用户看完能不能"马上用"？

> 有一个"不确定" → 先别发

**✅ 2. 有没有"本站独有的东西"（最关键）**

必须至少包含一个：
- 本站游戏库数据（折扣率/评分/礼包）
- 真实使用对比（盒子A vs 盒子B 具体数据）
- 明确观点（不是总结别人，是我们的判断）

```
❌ 错误："根据研究…很多人认为…这款游戏不错"
✅ 正确："我们测试了XX盒子，首充折扣是0.1折，比官服省了75元"
```

**✅ 3. 是否避免拼凑内容**

检查：
- 有没有只是把多个网站内容合并？
- 有没有只是换句话说（洗稿）？

> 是 → 直接重写

**✅ 4. 内容深度够吗？**

快速标准：
- 是否比 Google 前3名更有用？
- 是否有：步骤 / 示例 / 对比 / 注意事项

> 只是"解释概念" → HCU 会直接干掉

**✅ 5. 有没有废话段落**

删掉这些：
- "在当今数字时代…"
- "越来越多的人开始…"
- 空洞总结、无信息量的开头

**✅ 6. 标题不过度 SEO**

```
❌ 错误："2026最佳XXX终极指南（超详细+免费）"
✅ 正确：直接说明解决什么问题，越自然越好
```

**✅ 8. 不要关键词堆砌**

#### 方案C：AI指纹破除（避免被 SpamBrain 统计识别）

**问题根源：** 用同一套 Prompt 模板批量生产 → 文章结构/语气/句式高度相似 → 可被统计指纹识别

**执行规则：**

```
1. Prompt 多样化
   - 维护 3~5 套不同的 Prompt 模板（问答型/分析型/对比型/叙事型/建议型）
   - 同一类文章轮换使用，不连续使用同一模板
   - 禁止固定 Prompt 开头（如"作为一个游戏玩家..."）

2. 结构多样化
   - 文章格式轮换：Q&A格式 / 总结-展开 / 时间线 / 对比表格 / 数字列表
   - 正文长度变化：600字 / 900字 / 1200字 混排，不要全部相同长度

3. AI 禁用词/句型（写入 Prompt 约束）
   - "值得注意的是" / "此外" / "总的来说" / "综上所述"
   - "作为玩家" / "在游戏体验中" / "不得不说"
   - 固定格式总结段（每篇结尾要求不同）
```

#### 方案E：半自动写作流程 + 内容安全评分标准

> 这是把 AI 用对的核心：**人负责思考和经历，AI 负责补充和表达**

**一、标准文章骨架模板（适配本站游戏/盒子类内容）**

```
标题规范：【具体问题】+【解决结果】（有结果导向，不夸张，不标题党）

示例：
❌ "2026最佳游戏盒子终极指南（超详细+免费）"
✅ "XX游戏用哪个盒子最划算？我对比了3款，差价最高75元"
```

**开头（必须有"人味"）：**
```
我在玩XX游戏的时候，遇到了一个问题：XXX
试了很多方法（包括XXX），大部分没用。
后来发现了[盒子名]，实测折扣是[N折]，比官服省了[X元]。
这篇文章直接讲清楚怎么用。
```

**核心内容结构（每段必须带经验）：**
```
方法1：XXX
  我是怎么用的：（具体步骤）
  踩过的坑：（失败/问题）
  适合什么情况：（用户判断条件）

方法2：XXX（必须有对比）
  和方法1的区别：（直接对比，有数据）
  优点 / 缺点：
  我的实际建议：（给明确结论）
```

**反常识段落（加分项，必加）：**
```
很多人会以为：XXX
但实际情况是：XXX（本站数据支撑）
我的结论是：XXX
```

**结尾（不废话，直接给行动指引）：**
```
如果你只记住一件事：→ XXX（最关键结论）
新手：→ 建议从[方法X]开始
有基础：→ 直接用[方法X]
```

**二、AI 正确使用方式（三种安全用法）**

```
✅ 用法1：让 AI 写结构，你填内容
   Prompt："帮我整理一篇关于[XX盒子 vs 官服]的文章结构，包含方法/对比/常见错误"
   然后你加：本站游戏折扣数据 + 真实使用经验 + 你的观点

✅ 用法2：让 AI 改写你自己的内容
   Prompt："这是我写的一段内容，帮我优化表达，让它更清晰：[你的原文]"
   → 内容还是你的，AI只是优化表达

✅ 用法3：让 AI 补充细节
   Prompt："帮我补充这个步骤的细节：[你已有内容]"
   → 骨架是人写的，AI 填充说明

❌ 绝对不要这样用：
   "帮我写一篇XX文章" → 直接废掉（典型垃圾内容）
   批量生成几十篇同模板 → 整站足迹暴露
```

**三、半自动写作流程（比例是关键）**

```
Step 1：你写核心（占 20%）
  → 折扣数据 / 使用经验 / 你的观点 / 本站案例

Step 2：AI 扩展（占 50%）
  → 补结构 / 补说明 / 优化表达

Step 3：你再改（占 30%）
  → 加人话 / 加判断 / 删废话 / 加内链

原则：人的贡献 ≥ AI 的贡献
```

**四、内容安全评分标准（发布前自评）**

| 评级 | 特征 | 结果 |
|------|------|------|
| ✅ 安全 | 有个人/本站经验 + 有数据对比 + 有明确观点 + 结构清晰 + 不是拼凑 | 基本不会被 HCU 打 |
| ⚠️ 危险 | 大量AI生成 + 无本站数据 + 泛泛而谈 + 跟别人差不多 | 很容易掉排名 |
| ❌ 高危 | 批量生成+不改 / 洗稿 / 拼接 / 无任何价值 | 可能拖垮整站 |

> 核心原则：**AI 可以帮你写，但不能替你"思考和经历"**

- 多个关联站点互相 dofollow 链接 = Google 定义的 Link Scheme（链接图谱作弊）
- 被识别后：主站可能因外链图谱异常**反向降权**

## 漏洞 #4 ✅ 子站转化路径优化（已采用替代方案）

**状态：✅ 已解决（2026-04-18）**

**原问题描述：**
- 子站当前转化路径：子站读者 → 点击CTA → 跳转主站品类页 → 找盒子 → 点推广链接 → 注册
- 这是 4 步漏斗，每增加一步流失 40~60%

**已实施的优化：**
- 子站 DownloadBox 和 ArticleLayout 的 CTA 按钮现在**直接链接到主站 `/boxes` 页面**
- 缩短为 3 步漏斗：`子站文章 → 主站/boxes → 点推广链接 → 注册`

**为何不链接到游戏页：**
- 链接到游戏页需要推荐系统决定推荐哪个游戏
- 链接到 `/boxes` 页面让用户可以自由选择或搜索想玩的游戏，体验更好
- 所有外链已添加 `rel="nofollow noopener noreferrer"`，符合链接规范
